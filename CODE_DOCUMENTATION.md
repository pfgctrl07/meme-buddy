# Meme Buddy Code Documentation

## Overview

`Meme Buddy` is implemented as an Expo / React Native app with screens, reusable components, theme modules, mock data, and a backend-ready service / hook layer.

It is a mobile-first UI prototype with:

- Dashboard
- Create / Join Trend
- Event Detail
- Trend Discovery
- Alerts
- Leaderboard
- Profile

The current app uses local dummy data and local React state. It is ready to be connected to a backend by replacing the static data objects and local view state with API-driven data fetching and mutation calls.

## Project Files

- `App.js`
  Root app shell, navigation state, and screen composition.

- `src/screens/*`
  App screens split by feature.

- `src/components/*`
  Reusable UI components.

- `src/theme/*`
  Shared theme constants and styles.

- `src/data/mockData.js`
  Local preview fallback data.

- `src/config/api.js`
  API base URL setup using `EXPO_PUBLIC_API_BASE_URL`.

- `src/services/*`
  Domain service modules for backend requests.

- `src/hooks/*`
  Data-loading hooks that use services with mock fallback.

- `app.json`
  Expo app configuration.

- `package.json`
  Dependencies and scripts for Expo / React Native web preview.

- `dist/`
  Web export build output generated by Expo.

## Current Architecture

The app is now organized into:

1. Root shell
2. Theme + styles
3. Mock data fallback
4. Services
5. Hooks
6. Screen components
7. Reusable UI components

Main app state in `App.js`:

- `activeTab`
  Controls which screen is shown.

- `formTab`
  Controls `create` vs `join` mode in the create screen.

- `eventMetricTab`
  Controls `mentions` vs `volume` graph mode in the event detail screen.

- `discoveryFilter`
  Controls discovery sorting mode.

Data fetching now comes from:

- `useDashboardData()`
- `useTrendData()`
- `useDiscoveryData()`
- `useAlertsData()`
- `useLeaderboardData()`
- `useProfileData()`

These hooks call services and automatically fall back to local mock data when `EXPO_PUBLIC_API_BASE_URL` is not configured.

## Recommended Frontend Integration Plan

### Step 1: Move dummy data into a service layer

These files now exist:

- `src/services/dashboard.js`
- `src/services/trends.js`
- `src/services/alerts.js`
- `src/services/discovery.js`
- `src/services/leaderboard.js`
- `src/services/profile.js`

Shared API utilities:

- `src/config/api.js`
- `src/services/http.js`

Hooks:

- `src/hooks/useAsyncResource.js`
- `src/hooks/useAppData.js`

The services call the backend when `EXPO_PUBLIC_API_BASE_URL` is set and fall back to mock data otherwise.

Auth support:

- `src/services/http.js` now supports bearer auth.
- Token sources:
  - `EXPO_PUBLIC_API_TOKEN`
  - `setAuthToken(token)` at runtime

Example:

```js
export async function fetchDashboard() {
  const response = await fetch(`${API_BASE_URL}/api/dashboard`);
  if (!response.ok) throw new Error("Failed to load dashboard");
  return response.json();
}
```

### Step 2: Point the app at your backend

Set:

- `EXPO_PUBLIC_API_BASE_URL=https://your-api-domain.com`
- `EXPO_PUBLIC_API_TOKEN=your-token-if-needed`

Then the hooks will start using the real backend instead of mock data.

### Step 3: Add loading and error states

For each screen add:

- loading skeleton or loading text
- error message
- retry action

### Step 4: Add auth and user session

Recommended additions:

- auth token storage
- current user context
- protected API calls

## What the Backend Team Needs to Know

1. The UI expects event-level analytics and prediction metadata together.
2. Prediction is not just bullish/bearish; it also includes trust and confidence.
3. Alerts are a first-class feature and should be queryable independently.
4. Discovery feed should support server-side sorting/filtering.
5. Event detail needs both summary stats and timeseries arrays.
